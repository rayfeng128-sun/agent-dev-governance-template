# AGENTS.md

本文件用于约束 `references/Harness/` 所服务的<项目名称>治理体系。作用域为仓库根目录及所有子目录；若下级目录存在更近的 `AGENTS.md`，以更近文件为准。

## 工程基线

- 当前项目阶段为 `prototype-to-code`，交付目标是从原型仓库演进到可真实使用、可持续扩展的前后端分离系统。
- 当前主开发基线为 `frontend/` 下的 <前端技术栈> 前端应用。
- 当前主后端基线为 `backend/` 下的 <后端技术栈 API>，开发环境允许使用 demo profile 与 seed/fixture 数据承载测试数据，但接口、权限、状态流转和持久化边界必须按正式系统方式设计。
- `app/` 原生 HTML/CSS/JavaScript 与 `server/` 最小 Node 服务仅作为 legacy fallback，不再作为新页面和新接口的默认实现路径。
- 产品与页面基线以根目录 `AGENTS.md` 指定的 `docs/product/` 正式产品文档为准；Obsidian 主工作区位于 `<Obsidian 项目路径>/`。
- 在真实数据库和外部系统尚未全部接入前，允许使用受控 Mock、demo seed 和显式注释标记未确认字段，但这些手段只用于过渡，不构成长期交付标准。

## 快速入口

- agent 开始任务时，优先阅读 `references/Harness/QUICKSTART.md`。
- 只有任务触发完整闭环升级条件时，才继续阅读本文件、相关 `rules/`、角色技能和具体变更目录。
- 最终交付说明必须写明本次采用“轻量流程”还是“完整闭环流程”，并附验证依据。
- `PLANS.md` 作为当前阶段执行看板，必须在以下情况同步更新：任务完成后状态变化、阶段顺序变化、完成度判断变化、或新需求已写好且用户要求更新计划。

## references 目录职责

`references/Harness/` 是当前仓库的工程治理中心，采用五层结构：

1. `AGENTS.md`
   定义仓库级总约束、当前阶段技术边界和协作方式。
2. `rules/`
   定义工程结构、编码要求、开发流程、变更管理等强规则。
3. `agents/`
   定义角色边界，回答“谁负责什么”。
4. `skills/`
   定义动作模板，回答“每个角色如何执行”。
5. `changes/`
   定义版本化变更目录，回答“需求如何追踪、验证、交付”。

## 总体协作约束

- 当前阶段需求定义默认以用户故事和端到端闭环为主，以原型页和界面说明为辅。
- 所有页面结构、命名、状态语义和主操作必须服从 Obsidian 原型主稿，不得自行发明业务规则。
- 所有结论必须附带验证依据，不允许只写“已完成”而不说明如何验证。
- 对尚未确认的接口字段、状态流转、后端契约，必须用显式占位、受控测试数据、注释或 `api-contract.md` 说明，不允许隐式猜测。
- 前端新增或重写的源码默认优先采用 TypeScript，入口、路由、共享状态和公共工具优先使用 `.ts` 或 `<script setup lang="ts">`。
- 当前阶段默认以“可用系统闭环”为目标选择实现方式；低风险变更可走轻量流程，但不得把“演示可见”替代为“功能完成”。
- 开发必须遵循 `Plan -> Review -> Grooming -> Design -> Coding -> Testing -> Documentation` 的顺序；低风险任务可以简化产物，但不能跳过前置确认。
- 禁止直接重构整个系统、一次性修改多个无关模块，或删除已有功能后再补说明。
- 所有设计、流程和架构变更都必须同步到 `docs/`，并回写到当前 Obsidian 主工作区 `<Obsidian 项目路径>/`。
- 每次任务完成、阶段变化或需求更新且影响执行状态时，必须同步更新 `PLANS.md`。

## 两档交付门禁

### 轻量流程

适用于页面表现、样式、小型组件抽取、纯文档优化、局部非业务性重构等低风险变更。

- 需求记录仍应先写清用户故事、故事边界和最小可用闭环。
- 必须在 `references/Harness/changes/<version>/<change-id>/change.md` 记录目标、范围、原型来源、影响、验收、验证和风险。
- 必须执行与变更相关的最小验证，例如 `npm test`、`npm --prefix frontend run build` 或页面冒烟。
- 若执行过程中发现接口、数据库、权限、状态持久化或跨模块流程影响，必须升级为完整闭环流程。

### 完整闭环流程

适用于接口契约、数据库、权限模型、多页面流程、跨模块状态流转、发布级交付或架构调整。

- 完整闭环需求默认围绕一个可独立验收的用户故事或清晰的故事切片组织，而不是围绕单个原型页组织。
- 必须按 `Change-Management-Rule.md` 创建完整需求目录。
- 涉及接口变化时，`api-contract.md` 必须先更新。
- 涉及数据库或持久化结构时，`db-migrations.sql` 与 `rollback.sql` 必须同时存在。
- 涉及多页面流程、权限或状态流转时，必须补齐 `flow.md` 与 `impact.md`。
- 未完成 `smoke-checklist.md` 和 `summary.md`，不得视为完整交付闭环。

## 角色约束

当前治理体系采用五类角色：

- `requirement-analyst`
  负责接收原型、需求和背景材料，产出结构化需求、页面映射和流程说明。
- `solution-architect`
  负责方案设计、边界判断、影响分析和规则修订建议。
- `frontend-developer`
  默认负责 `frontend/` 下 Vue 页面、共享 UI、状态模型和 API 消费实现；仅维护 legacy fallback 时才进入 `app/`。
- `backend-developer`
  默认负责 `backend/` 下 <后端技术栈 API>、真实业务逻辑、测试数据承载方式与持久化边界；仅维护 legacy fallback 时才进入 `server/`。
- `fullstack-integrator`
  负责联调、冒烟、文档齐套性检查与最终交付收口。

## 原型产物约束

- Obsidian 文档是当前阶段页面型需求的正式设计基线。
- 原型产物用于说明界面、交互和信息结构，不替代用户故事、价值和端到端闭环定义。
- 轻量流程可在 `change.md` 中直接记录原型来源与页面映射。
- 完整闭环流程中，若已有 Stitch 原型，在 `stitch-link.md` 中记录链接、版本说明和需求映射。
- 完整闭环流程中，若暂无 Stitch 链接，也必须在 `stitch-link.md` 中明确写明“当前使用 Obsidian 高保真原型作为替代基线”。
- `prototype.md` 用于描述页面结构、关键交互、状态说明和字段语义。
- `flow.md` 用于描述用户路径、业务流转、异常分支和前后置条件。

- 当任务完成情况、阶段判断或需求优先级变化影响当前执行看板时，必须同步更新根目录 `PLANS.md`。
- 不允许把不适配当前阶段的外部技术栈基线直接照搬进规则。

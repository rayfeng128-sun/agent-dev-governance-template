# 开发流程规则

## 1. 目标

本规则定义<项目名称>仓库从需求登记到交付归档的执行流程。当前阶段采用“两档治理”：默认走轻量流程处理低风险变更，命中高风险条件时升级为完整闭环流程。无论轻量还是完整流程，需求定义默认都应以“用户故事和端到端可用闭环”为主，以原型页和界面说明为辅。

## 2. 流程选择

### 2.1 轻量流程

适用于低风险、可快速验证的变更：

- 页面表现、布局、样式和文案调整。
- 非业务性静态信息调整和前端壳子优化。
- 小型共享组件抽取，不改变业务状态流转。
- 纯治理说明、文档入口和模板优化。

必需产物：

- `references/Harness/changes/<version>/<change-id>/change.md`

最小流程：

1. 在 `change.md` 写明用户故事、目标、范围、原型来源、影响、验收、验证和风险。
2. 按当前主线在 `frontend/`、`backend/` 或治理文件中实施最小变更。
3. 执行与变更范围匹配的验证命令。
4. 在最终答复中说明采用轻量流程与验证结果。

### 2.2 完整闭环流程

适用于高风险、跨模块或发布级变更：

- 新增或修改后端 API、响应结构、错误码或接口语义。
- 新增或修改数据库、持久化结构、迁移脚本或回滚策略。
- 修改权限模型、角色能力、导航准入或关键操作授权。
- 修改多页面流程、核心状态流转、跨模块数据链路。
- 发布级交付、架构调整或需要正式审计的需求。

完整流程共十个阶段：

1. 需求登记
2. 需求澄清与结构化整理
3. 原型基线登记
4. 流程说明补齐
5. 影响分析
6. 架构决策
7. 开发实施
8. 变更脚本补齐
9. 联调与冒烟
10. 交付总结

## 3. 轻量流程要求

### 3.1 需求记录

必需产物：

- `change.md`

要求：

- 明确本次变更目标、范围和非目标。
- 页面型变更默认先说明“谁、在什么场景下、要完成什么、获得什么价值”。
- 页面型变更必须记录 Obsidian 或 Stitch 原型来源；无页面影响时写明“不涉及页面原型”。
- 明确本次变更对应的最小故事闭环和不包含的故事边界。
- 明确影响范围，包括前端、后端、测试数据、权限、配置、测试和真实使用路径中实际受影响的部分。
- 明确验收标准和已执行验证。

### 3.2 升级检查

轻量流程执行过程中，若发现以下任一情况，必须升级为完整闭环流程：

- 需要新增或修改后端 API。
- 需要新增或修改数据库或持久化结构。
- 需要调整权限、角色、登录态或操作准入。
- 需要改变多页面用户路径或核心状态流转。
- 需要把未确认业务规则固化进代码。

## 4. 完整流程阶段定义

### 4.1 需求登记

必需产物：

- `request.md`

要求：

- 明确用户故事、业务目标、范围、非目标、端到端主路径和验收标准。
- 未创建需求目录或未完成 `request.md`，不得开始完整流程设计和实现。

### 4.2 需求澄清与结构化整理

必需产物：

- `request.md`

要求：

- 对原始需求、原型页面、截图和业务说明做结构化整理。
- 明确用户角色、触发场景、目标、业务价值、故事边界、验收标准和未决问题。

### 4.3 原型基线登记

必需产物：

- `prototype.md`
- `stitch-link.md`

要求：

- 页面型或交互型需求必须登记原型基线。
- 原型基线用于说明界面和交互，不替代用户故事和业务闭环定义。
- 若存在 Stitch 原型，`stitch-link.md` 必须记录链接、版本说明和需求映射。
- 若当前仍以 Obsidian 高保真原型为准，必须在 `stitch-link.md` 中显式写明替代基线来源。
- 若需求为纯后端或纯治理型，可在 `stitch-link.md` 中说明“本需求无需页面原型”并解释原因。

### 4.4 流程说明补齐

必需产物：

- `flow.md`

要求：

- 明确用户路径、业务流转、异常分支、前后置条件以及故事起点/终点。
- 涉及多页面流程、权限或状态流转时，未完成 `flow.md` 不得进入开发。

### 4.5 影响分析

必需产物：

- `impact.md`
- 涉及接口变化时，提前创建 `api-contract.md`

要求：

- 明确影响前端、后端、测试数据、权限、配置、测试和真实使用路径的范围。
- 明确本次实现是否构成独立可用故事，若不是，必须说明拆分原因和剩余缺口。
- 涉及跨模块开发时，未完成 `impact.md` 不得进入开发。

### 4.6 架构决策

必需产物：

- 影响分析后的方案决策结论
- 必要时更新 `rules/` 或补充需求目录中的设计说明

要求：

- 明确模块边界、共享抽取点、接口边界和真实后端接入边界。
- 如发现现有规则无法覆盖本次变更，必须先修订规则，再进入实现。

### 4.7 开发实施

必需产物：

- 代码变更
- 接口、测试数据或 seed 更新
- 配置变更说明

要求：

- 所有开发都必须基于本次流程所需产物和最新契约产物。
- 涉及接口变化时，必须先更新 `api-contract.md`，再开始前后端实现。

### 4.8 变更脚本补齐

必需产物：

- `db-migrations.sql`
- `rollback.sql`

要求：

- 若涉及数据库或持久化结构变化，必须同时提供正向脚本和回滚脚本。
- 若完整闭环流程中无数据库变化，也必须在两个文件中说明“无数据库变更”。

### 4.9 联调与冒烟

必需产物：

- `smoke-checklist.md`

要求：

- 覆盖关键业务链路、异常链路、权限链路和发布前核验点。
- 页面型需求应以 Obsidian 原型或 Stitch 原型作为联调与冒烟对照基线。
- 未完成冒烟清单勾检，不得视为完整交付闭环。

### 4.10 交付总结

必需产物：

- `summary.md`

要求：

- 记录最终实现、验证结果、用户故事闭环达成情况、风险说明、后续接后端边界和遗留事项。
- `summary.md` 未完成，不得进入版本归档或发布结论。

## 5. 强制门禁

以下门禁不可绕过：

1. 轻量流程未创建 `change.md`，不得标记为完成。
2. 完整流程未创建 `request.md`，不得开始完整实现。
3. 需求若未定义用户故事或最小闭环，不得以“页面已完成”替代需求完成。
4. 页面型完整流程未完成 `prototype.md` 与 `stitch-link.md`，不得进入页面开发。
5. 涉及多页面流程、权限或状态流转时，必须完成 `flow.md` 与 `impact.md`。
6. 涉及接口变化时，`api-contract.md` 必须先更新。
7. 涉及数据库或持久化结构变化时，`db-migrations.sql` 与 `rollback.sql` 必须同时存在。
8. 完整流程中 `smoke-checklist.md` 未完成，不得标记为已验收。
9. 完整流程中 `summary.md` 未完成，不得标记为已交付。

## 6. 对 agents 的强制要求

- 所有 agent 默认先阅读 `references/Harness/QUICKSTART.md`。
- 命中完整闭环升级条件时，再阅读仓库根 `AGENTS.md`、[references/Harness/AGENTS.md](../AGENTS.md) 和相关 `rules/`。
- 所有 agent 必须按选定流程产物推进，不得只改代码不补记录。
- `requirement-analyst` 必须按变更风险选择轻量记录或完整需求整理。
- `solution-architect` 必须基于流程风险完成边界判断，必要时升级完整闭环。
- `frontend-developer` 默认依据 `frontend/` 主线开发页面；维护 legacy fallback 时才进入 `app/`。
- `backend-developer` 默认依据 `backend/` Spring Boot 主线开发接口；维护 legacy fallback 时才进入 `server/`。
- `fullstack-integrator` 必须检查本次选定流程的产物和验证是否齐全，不齐全时不得判定闭环。

## 7. 违规示例

以下行为均视为违规：

- 低风险变更没有 `change.md` 就声称完成。
- 高风险变更用轻量流程绕过接口、权限、数据库或跨模块影响分析。
- 把单个原型页面直接当作默认需求单位，却不说明对应用户价值和故事边界。
- 页面型需求没有原型来源记录就进入开发。
- 前端按猜测字段实现页面，未在 Mock、demo seed 或契约文档中记录假设。
- 改了接口或数据结构，但没有同步 `api-contract.md`。
- 冒烟未执行，就在总结中写“验证通过”。

# 编码规范

## 1. 目标

本规范用于统一当前仓库当前阶段的编码风格、过渡数据使用方式、注释边界和共享抽取原则。当前主线为 `frontend/` Vue 3 应用与 `backend/` <后端技术栈 API>。

## 2. 通用要求

- 默认使用 ASCII 字符编写代码；中文文档必须保持 UTF-8、表达清晰。
- 命名优先体现业务语义，而不是技术细节。
- 共享逻辑优先抽离到已有共享文件，避免复制粘贴。
- 注释仅用于解释不自明的边界、过渡假设或后续真实后端接入点。

## 3. 前端规则

- 新页面和新能力默认落在 `frontend/`，仅维护 legacy fallback 时才进入 `app/`。
- 前端结构默认遵循 `docs/architecture/code-structure-guidelines.md`，后续向 `shared + modules` 的混合业务域结构渐进收敛。
- 新增复杂业务能力优先落到 `modules/<domain>/`，不要继续堆到全局大页面、全局 API 文件或单个超大组件。
- `shared/` 只承载跨业务域通用的 HTTP、布局、组件、会话、权限和基础状态，不得放入某个业务域专属规则。
- 页面级视图只负责页面编排，不堆积过多业务判断。
- 共享布局、导航、状态标签、统计条、风险面板、表格样式优先复用。
- 所有未确认的字段、状态和接口返回必须通过受控测试数据、seed 或注释显式表达。
- 需要权限控制的主操作必须通过权限模型或会话模型判断，不允许散落魔法字符串。
- 页面应覆盖加载态、空态、错误态和权限态，且关键交互应能真实运行。
- 页面中禁止直接操作底层 HTTP 细节，应通过共享 API client 或业务 API 封装调用。

## 4. 后端规则

- 新接口和新契约默认落在 `backend/` Spring Boot 主线，`server/` 仅作为 legacy Node fallback。
- 后端结构默认遵循 `docs/architecture/code-structure-guidelines.md`，继续采用 Spring Boot 单体分模块、业务域分包结构。
- Spring controller 保持 HTTP 边界清晰，不承载复杂业务组装。
- Controller 只能调用 service，不直接访问 repository、store、job store 或 `DemoBackendService`。
- service 层负责接口级业务数据组装，并明确测试数据、过渡逻辑或真实持久化边界。
- service 优先通过 store / job store 访问数据边界，不把 demo/local 分支和持久化细节散落在多个方法中。
- repository 只负责持久化访问，不承载状态流转、跨模块业务流程或响应拼装。
- `common`、`runtime`、`security` 等公共边界只能承载通用基础设施、运行态或安全能力，不得放入业务域专属规则。
- 接口返回结构优先保持稳定；若当前阶段暂无统一响应包裹，必须在契约文档中说明。
- 若接口仍处于过渡实现阶段，必须在服务层或契约文档中标记其临时性质与收敛计划。

## 5. 文档与变更规则

- 所有实现都必须按风险进入对应 `changes/` 目录：低风险使用 `change.md`，高风险使用完整闭环文件。
- `api-contract.md` 中的“无变化”也属于有效结论，禁止留空。
- 只有完整闭环流程或涉及数据库/持久化结构时，才必须提供 `db-migrations.sql` 和 `rollback.sql`。
- 最终交付说明必须写明采用轻量流程还是完整闭环流程，并附验证依据。

## 6. 禁止事项

- 禁止在没有基线依据时发明业务规则。
- 禁止在页面中直接操作底层 HTTP 细节。
- 禁止 Controller 直接访问 repository 或 `DemoBackendService`。
- 禁止把复杂业务继续沉积到全局页面、全局 API 文件、巨型 service 或公共层。
- 禁止把多个独立需求混在同一变更目录。
- 禁止用轻量流程绕过接口、数据库、权限或跨模块流程影响。
- 禁止用“后面再补文档”绕过当前阶段产物要求。

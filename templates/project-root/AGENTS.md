## Engineering Rules

- 默认沿 `frontend/` <前端技术栈> 与 `backend/` Spring Boot 主线推进；`app/` 与 `server/` 仅作为 legacy fallback。
- 需求定义默认以用户故事为主、以原型页为辅；页面用于说明界面与交互，不作为默认需求切分单位。
- 默认优先实现真实前后端闭环、真实状态流转和可保存的业务结果，不再把“仅可演示”作为主要交付标准。
- Mock 数据只允许用于未完成接口、复杂外部依赖或高风险能力的短期过渡，不得长期作为业务真相来源。
- demo seed / fixture 只作为开发与测试环境的真实数据承载方式，不应替代正式接口契约、权限校验和持久化边界设计。
- 页面必须同时考虑刷新后状态一致性、失败处理、权限限制和后续真实数据落库路径。
- 页面结构、命名、状态语义应严格服从原型主稿。
- 优先抽离共享布局、导航、状态标签、统计条、风险面板、表格样式。
- 新代码默认遵循 `docs/architecture/code-structure-guidelines.md` 的业务域分包结构规范，并同时遵循 `references/Harness/rules/Code-Rule.md` 的调用边界。
- 前端后续向 `shared + modules` 混合业务域结构渐进收敛；后端继续按 Spring Boot 单体分模块、业务域分包结构推进。
- 结构迁移必须小步进行，优先以单个用户故事、单条主链路或单个业务域切片推进，禁止无验证口径的一次性大重构。
- 不要在没有依据时发明新业务规则。
- 对尚未确认的后端字段、状态流转、接口契约，用显式占位、契约文档和注释标记，不得把猜测逻辑直接固化为正式实现。
- 前端新代码默认优先使用 TypeScript，新增或重写的共享模块、入口、路由和状态层优先采用 `.ts` 或 `<script setup lang="ts">`。

## Development Principles

- 先规划后开发：任何代码修改前，先明确需求、边界、影响和验证方式。
- 先输出设计方案：涉及业务、流程、接口或结构调整时，先形成可评审的设计或变更说明，再进入实现。
- 未确认方案不得写代码：接口字段、状态流转、页面行为、数据持久化未确认时，只允许做显式占位、契约说明或受控 Mock。
- 小步迭代：优先拆成单个用户故事、单条主链路或单个可验收切片推进，避免一次性铺开。
- 每个功能必须可测试：交付必须能用测试、冒烟或明确操作步骤验证，不接受不可复核结果。
- 每次修改必须更新文档：凡是影响需求、流程、结构、计划或交付结论的改动，都要同步更新对应文档与 `PLANS.md`。

## Development Flow

标准开发顺序为：

`Plan` -> `Review` -> `Grooming` -> `Design` -> `Coding` -> `Testing` -> `Documentation`

- `Plan`：确认目标、范围、优先级和完成标准。
- `Review`：复核现有约束、原型和相关文档，避免重复造轮子。
- `Grooming`：将宽泛需求收敛为可执行任务和最小闭环。
- `Design`：输出方案、接口、影响范围和验证方式。
- `Coding`：按已确认方案实现，避免边写边改需求。
- `Testing`：用测试或冒烟确认结果可复现、可回归。
- `Documentation`：同步更新需求、计划、变更记录和必要的设计文档。

## Forbidden Actions

- 禁止直接重构整个系统，必须先拆分范围并分阶段推进。
- 禁止一次性修改多个无关模块，除非它们属于同一个已确认闭环且已拆分验证。
- 禁止删除已有功能，除非有明确替代方案、迁移路径和回退说明。

## Done Definition

一个开发任务完成时，至少满足：

- 至少交付一个可真实使用的用户故事闭环或清晰的可用故事切片
- 与原型页面关系清晰
- 主路径、关键状态、异常路径和主要操作可运行
- 前后端真实联通，核心数据可被查询、保存并在刷新后保持一致
- 关键操作具备基础权限控制、参数校验和失败提示
- 本地开发环境可以完整运行，而不只是静态预览
- 后续扩展数据库、权限、外部系统和部署方案的边界清楚

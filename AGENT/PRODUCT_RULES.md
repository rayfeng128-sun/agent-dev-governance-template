# Product Rules

## 目标

产品规则用于约束 agent 如何理解需求、拆分用户故事、确认范围和维护产品事实来源。

## 事实来源

- 每个项目必须明确产品事实来源，例如 `docs/product/`、PRD、用户流程、原型或业务手册。
- 当代码与文档不一致时，不得自行猜测业务规则，必须显式标记差异并请求确认或更新文档。
- 原型页、截图和 UI 草稿只能说明界面和交互，不替代用户故事、业务价值和端到端闭环。

## 需求定义

默认使用用户故事表达需求：

```text
作为 <用户角色>
在 <业务场景> 下
我希望 <完成某个动作或目标>
从而 <获得业务价值>
```

每个需求至少说明：

- 用户角色
- 触发场景
- 业务目标
- 包含范围
- 非目标
- 主路径
- 异常路径
- 验收标准
- 验证方式

## 颗粒度

- 一个任务应围绕一个用户故事、一个故事切片或一个清晰技术边界。
- 不要把 Epic 当作开发任务，例如“全面重构”“统一模块边界”“完善系统能力”。
- 超过 0.5 到 2 天仍无法关闭的任务，应继续拆分。
- 任务必须有明确退出标准，不得把“持续优化”写成完成条件。

## 原型与页面

- 页面用于辅助说明信息结构、交互和状态，不作为默认需求切分单位。
- 页面型需求必须说明对应用户故事和故事边界。
- 如果实现与原型不一致，必须记录原因、影响和后续处理方式。

## Done Definition

一个产品需求完成时至少满足：

- 交付了一个可独立验收的故事闭环或故事切片。
- 主路径、关键异常路径和主要状态可运行。
- 用户可理解当前结果和下一步操作。
- 核心数据刷新后保持一致，或明确说明仍是过渡数据。
- 验收标准和实际验证结果已记录。

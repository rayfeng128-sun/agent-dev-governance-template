# Test Rules

## 目标

测试规则用于约束 agent 如何证明变更可用、可回归、可验收。

## 测试分层

按风险选择最小有效验证：

| 变更类型 | 最小验证 |
|---|---|
| 文档 / 治理规则 | 文档链接检查、治理测试或 grep 检查 |
| 前端页面 / 组件 | 前端构建、组件测试、页面冒烟 |
| 后端 service / API | 单元测试、契约测试、接口冒烟 |
| 数据库 / 持久化 | migration 检查、repository 测试、回滚说明 |
| 权限 / 安全 | 权限路径测试、拒绝路径测试 |
| 跨模块流程 | 主路径冒烟、异常路径冒烟、回归测试 |

## 测试原则

- 先明确验证目标，再选择测试命令。
- 高风险变更必须补自动化测试或清晰冒烟清单。
- 修 bug 时优先补失败测试，再修实现。
- 涉及 API、数据库、权限、状态流转时，不能只做静态检查。
- 测试失败时必须说明失败原因，不得包装成通过。

## 冒烟要求

冒烟清单至少覆盖：

- 主路径是否可完成。
- 核心异常路径是否可见。
- 权限拒绝是否符合预期。
- 刷新后状态是否一致。
- 数据是否能查询、保存或回放。

## 完成标准

任务完成前必须记录：

- 实际运行的命令。
- 通过或失败结果。
- 未运行的验证项及原因。
- 仍需补充的测试缺口。

## 禁止事项

- 禁止未验证就声称完成。
- 禁止只跑无关测试充当验证。
- 禁止忽略失败测试。
- 禁止修改测试来迎合错误实现，除非测试本身已确认过时。

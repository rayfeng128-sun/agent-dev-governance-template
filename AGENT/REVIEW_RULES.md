# Review Rules

## 目标

Review 规则用于约束 agent 如何进行需求 review、架构 review、代码 review 和交付 review。

## Review 顺序

推荐顺序：

```text
Requirement Review -> Architecture Review -> Code Review -> Test Review -> Documentation Review
```

## 需求 Review

检查：

- 是否有明确用户故事。
- 是否有范围和非目标。
- 是否有主路径、异常路径和验收标准。
- 是否把页面误当成完整需求。
- 是否存在未确认业务规则。

## 架构 Review

检查：

- 是否符合项目目录职责。
- 是否跨越多个无关模块。
- 是否引入不必要抽象。
- 是否破坏依赖方向。
- 是否需要更新架构文档。

## 代码 Review

优先找问题，不先写总结。

重点检查：

- 行为回归。
- 权限或安全问题。
- 数据一致性问题。
- 错误处理缺失。
- 测试缺失。
- 过度设计。
- 与既有模式不一致。

发现问题时应说明：

```text
严重级别
文件和位置
问题描述
影响
建议修复方向
```

## 测试 Review

检查：

- 测试是否覆盖本次真实风险。
- 是否有主路径和异常路径。
- 是否存在只测 mock、不测真实边界的问题。
- 是否有失败测试未处理。

## 文档 Review

检查：

- `PLANS.md` 是否更新。
- 需求或变更目录是否更新。
- API、数据库、权限、流程变更是否有对应文档。
- 最终答复是否包含验证结果。

## 禁止事项

- 禁止把“能运行”当成 review 通过。
- 禁止只写优点不列风险。
- 禁止用宽泛描述替代文件和位置。
- 禁止没有验证证据就批准交付。

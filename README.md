# Agent 开发治理框架抽取

本文档从当前项目中抽取一套可复用的 agent 开发规则与协作框架，供后续复制到其他项目时改造使用。本文不绑定某个具体业务，也不绑定具体技术栈。

## 1. 适用场景

这套框架适用于以下项目：

- 需要由 AI agent 长期协助开发、重构、测试和文档维护。
- 项目会持续演进，而不是一次性脚本或短期原型。
- 存在产品文档、架构文档、计划文档和代码实现之间容易脱节的问题。
- 需要控制 agent 不要直接大范围重构、猜测业务规则或跳过验证。

不适合直接套用的场景：

- 一次性实验脚本。
- 没有持续维护计划的小工具。
- 完全没有文档、计划和验收要求的探索项目。

## 2. 核心思想

### 2.1 入口先行

每个项目都应该有一个根目录 `AGENTS.md`，用于告诉 agent：

- 项目目标是什么。
- 当前阶段是什么。
- 哪些文档是事实来源。
- 当前主线目录和 legacy fallback 是什么。
- 开发原则、禁止事项和完成定义是什么。

推荐入口顺序：

```text
AGENTS.md
references/Harness/QUICKSTART.md
docs/README.md
PLANS.md
```

### 2.2 用户故事优先

需求定义默认以用户故事和端到端闭环为主，原型页、截图和页面清单只作为辅助说明。

推荐表达：

```text
作为 <用户角色>
在 <业务场景> 下
我希望 <完成某个动作或目标>
从而 <获得业务价值>
```

不要把“完成某个页面”直接当作需求完成。页面只能说明界面和交互，不能替代业务价值、状态流转和可验收结果。

### 2.3 两档流程

默认将 agent 开发流程分成两档：

| 流程 | 适用范围 | 最小产物 | 验证要求 |
|---|---|---|---|
| 轻量流程 | 文案、样式、小组件、低风险文档优化、局部非业务重构 | `change.md` | 最小相关测试或冒烟 |
| 完整闭环流程 | API、数据库、权限、多页面流程、核心状态流转、发布级交付 | 完整变更目录 | 契约、迁移、冒烟和总结齐全 |

轻量不是“不记录”，而是用更少的产物记录清楚目标、边界和验证。完整闭环也不是“形式主义”，而是给高风险变更留下可追踪的设计、影响和回退依据。

## 3. 标准开发原则

建议所有项目默认写入 `AGENTS.md`：

- 先规划后开发：任何代码修改前，先明确目标、边界、影响和验证方式。
- 先输出设计方案：涉及业务、流程、接口或结构调整时，先形成可评审方案。
- 未确认方案不得写代码：字段、状态、权限、数据持久化未确认时，只允许显式占位或契约说明。
- 小步迭代：优先拆成单个用户故事、单条主链路或单个可验收切片。
- 每个功能必须可测试：结果必须能用测试、冒烟或明确操作步骤复核。
- 每次修改必须更新文档：影响需求、计划、架构、流程或交付结论时，同步更新文档和计划。

推荐标准流程：

```text
Plan -> Review -> Grooming -> Design -> Coding -> Testing -> Documentation
```

每一步含义：

| 阶段 | 目标 |
|---|---|
| Plan | 确认目标、范围、优先级和完成标准 |
| Review | 复核现有代码、文档、原型和约束 |
| Grooming | 将宽泛需求拆成可执行的小任务 |
| Design | 输出方案、接口、影响范围和验证方式 |
| Coding | 按已确认方案实施 |
| Testing | 用测试或冒烟确认结果可复现 |
| Documentation | 同步需求、计划、变更记录和设计文档 |

## 4. 禁止事项

建议在所有项目中写成强规则：

- 禁止直接重构整个系统，必须先拆范围并分阶段推进。
- 禁止一次性修改多个无关模块，除非它们属于同一个已确认闭环。
- 禁止删除已有功能，除非有明确替代方案、迁移路径和回退说明。
- 禁止在没有事实来源时发明业务规则。
- 禁止接口、数据库、权限或跨模块流程变更仍走轻量流程。
- 禁止只改代码不补需求记录、契约、验证和计划。
- 禁止用“后面再补文档”绕过当前任务的交付要求。

## 5. 推荐目录结构

可复制的治理目录结构：

```text
<project-root>/
  AGENTS.md
  PLANS.md
  README.md
  docs/
    README.md
    product/
    architecture/
    process/
    archive/
  references/
    Harness/
      AGENTS.md
      QUICKSTART.md
      rules/
        Project-Arch.md
        Dev-Process-Rule.md
        Change-Management-Rule.md
        Code-Rule.md
      agents/
      skills/
      changes/
        README.md
        v-template/
        v0.1.0/
```

目录职责：

| 路径 | 职责 |
|---|---|
| `AGENTS.md` | 仓库级总约束，告诉 agent 当前项目怎么做 |
| `PLANS.md` | 当前执行看板，不只是愿望清单 |
| `docs/product/` | 产品需求、用户流程、页面和原型基线 |
| `docs/architecture/` | 系统架构、接口、数据、部署和边界设计 |
| `docs/process/` | 实施计划、阶段状态、AI spec/plan 和过程记录 |
| `references/Harness/` | agent 治理规则、角色、技能和变更目录 |
| `references/Harness/changes/` | 版本化需求变更和交付证据 |

## 6. 变更目录模板

### 6.1 轻量变更

适用于低风险、可独立验证的小变更。

```text
references/Harness/changes/<version>/<change-id>/
  change.md
```

`change.md` 建议包含：

| 字段 | 说明 |
|---|---|
| Goal | 本次变更目标 |
| User Story | 谁、在什么场景下、想完成什么、获得什么价值 |
| Scope | 包含和不包含什么 |
| Prototype Source | 页面型变更对应的原型来源；无页面影响时说明原因 |
| Story Boundary | 本次闭环到哪里，后续不包含什么 |
| Impact | 影响前端、后端、测试数据、权限、配置、测试和真实使用路径的哪些部分 |
| Acceptance Criteria | 如何判断完成 |
| Verification | 实际执行的验证命令或冒烟检查 |
| Risks | 遗留风险、未确认假设或升级条件 |

### 6.2 完整闭环变更

适用于 API、数据库、权限、多页面流程、核心状态流转和发布级交付。

```text
references/Harness/changes/<version>/<change-id>/
  request.md
  prototype.md
  stitch-link.md
  flow.md
  impact.md
  api-contract.md
  db-migrations.sql
  rollback.sql
  smoke-checklist.md
  summary.md
```

完整闭环的关键门禁：

| 触发条件 | 必需产物 |
|---|---|
| 新增或修改 API | `api-contract.md` |
| 新增或修改数据库 / 持久化结构 | `db-migrations.sql`、`rollback.sql` |
| 修改权限、角色、登录态或关键操作准入 | `flow.md`、`impact.md` |
| 修改多页面流程或核心状态流转 | `flow.md`、`impact.md`、`smoke-checklist.md` |
| 发布级交付或架构调整 | 完整十文件闭环 |

## 7. 任务颗粒度控制

不要把 Epic 当成任务。类似“统一模块边界”“全面重构”“完善系统能力”“收口全部主链”这类表述，更适合作为主题，而不是单个可执行任务。

一个可执行任务建议满足：

- 只围绕一个业务模块或一个清晰技术边界。
- 最好 0.5 到 2 天能完成。
- 有明确退出标准。
- 有明确验证命令或冒烟路径。
- 能独立更新计划状态。
- 不把“持续优化”写进完成标准。

推荐拆分方式：

| 大主题 | 拆分方式 |
|---|---|
| 统一后端模块边界 | 按模块拆：exam-plan、registration、score、ticket |
| 前端体验收口 | 按用户故事或页面主链拆 |
| 数据持久化升级 | 按聚合根或主流程拆 |
| 权限体系完善 | 按角色和操作准入拆 |
| 架构治理 | 按规则、目录、测试和文档入口拆 |

## 8. 文档同步策略

建议采用“双层保存”：

| 位置 | 用途 | 建议 |
|---|---|---|
| 独立模板项目 / 独立仓库 | 存放可复用框架、模板、规则、脚手架 | 推荐作为主来源 |
| Obsidian | 存放理解笔记、方法论、决策解释、项目复盘 | 推荐作为阅读和沉淀层 |
| 具体项目仓库 | 存放项目定制后的 `AGENTS.md`、`PLANS.md`、`references/Harness/` | 每个项目都应保留一份本地化版本 |

不建议只放 Obsidian。原因是 Obsidian 适合沉淀知识，但不适合作为 agent 每次执行时的强约束入口。agent 真正需要的是跟代码同仓的、可版本化的规则文件。

推荐做法：

1. 建一个独立的 `agent-dev-governance-template` 项目，作为可复用模板主来源。
2. 在 Obsidian 中建立一篇方法论笔记，链接到模板项目，并记录为什么这样设计。
3. 每个新项目复制模板目录，再把项目目标、技术栈、事实来源和验证命令替换成本项目自己的内容。

## 9. 新项目落地步骤

复制到新项目时，建议按这个顺序做：

1. 创建根目录 `AGENTS.md`，写清项目目标、阶段、事实来源、主线目录和禁止事项。
2. 创建 `PLANS.md`，只放可执行、可关闭、可验证的任务。
3. 创建 `docs/README.md`，说明产品、架构、过程、归档文档入口。
4. 创建 `references/Harness/QUICKSTART.md`，作为 agent 默认轻量入口。
5. 创建 `references/Harness/rules/`，放结构、流程、变更和编码规则。
6. 创建 `references/Harness/changes/v-template/`，放轻量和完整变更模板。
7. 为项目写最小验证命令，例如测试、构建、lint、冒烟。
8. 先用一个低风险需求跑通轻量流程，再用一个中等风险需求验证完整闭环流程。

## 10. 可复用检查清单

新项目启用前检查：

- `AGENTS.md` 是否说明项目目标、当前阶段和事实来源。
- `PLANS.md` 是否能反映任务完成度，而不是宽泛愿望。
- 是否明确主线目录和 legacy fallback。
- 是否明确“轻量流程”和“完整闭环流程”的升级条件。
- 是否明确 API、数据库、权限、多页面流程的门禁。
- 是否禁止直接大重构、一次性多模块修改和删除已有功能。
- 是否规定每个功能必须可测试。
- 是否规定任务完成后更新文档和计划。
- 是否有可运行的最小验证命令。
- 是否有变更目录模板。
